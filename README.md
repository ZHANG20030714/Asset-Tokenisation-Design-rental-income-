# Rental Income Tokenisation – RNT

This repository contains a minimal prototype implementation of a rental income tokenisation scheme designed for the UCL module *Decentralised Finance and Blockchain (IFTE0007)*. The project models how future rental cash flows from a pool of residential properties can be represented and distributed via fungible tokens on a public blockchain.

---

## 1. Concept Overview

### Economic Idea

- **Underlying asset**: Future net rental income generated by a defined pool of residential properties over a specified term (e.g. 5 years).
- **Token**: An ERC‑20 fungible token, `RNT` (“Rental Income Token”), where each token represents an identical pro‑rata claim on the net rental income pool.
- **Distribution**: Rental income is periodically remitted (in a stablecoin / reward token) to an on‑chain distribution contract, which allocates it to `RNT` holders according to their current token balances.

This prototype focuses on illustrating the **asset → token → cash flow distribution** mechanism, rather than full legal or production deployment.

---

## 2. Contracts

### 2.1 `RentToken.sol`

- Standard ERC‑20 token using OpenZeppelin.
- Represents fractional claims on the rental income pool.
- Key properties:
  - **Name**: `Rental Income Token` (configurable)
  - **Symbol**: `RNT` (configurable)
  - **Initial supply**: Minted entirely to the deployer (typically the SPV / issuer).
- Main functions:
  - `transfer`, `approve`, `transferFrom` (standard ERC‑20)
  - `totalSupply`, `balanceOf` (standard ERC‑20)

### 2.2 `RentDistributor.sol`

- Distributes periodic rental income (in a reward token, e.g. a stablecoin) to `RNT` holders.
- Uses a cumulative “reward per token” accounting model.
- Key elements:
  - `rentToken`: Address of the `RNT` ERC‑20 contract.
  - `rewardToken`: Address of the token used for payouts (e.g. test stablecoin).
  - `notifyRewardAmount(uint256 reward)`: Called by the owner (asset manager) to register a new rental income amount for distribution.
  - `earned(address account)`: View function showing how much a holder can claim.
  - `getReward()`: Allows a holder to claim their accrued rental income.

> Note: In a real deployment, the asset manager would first transfer `rewardToken` to the distributor contract, then call `notifyRewardAmount`.

You may optionally add a simple ERC‑20 `RewardToken` contract for local testing.

## 4. Deployment Instructions (Example with Remix)

1. **Prepare OpenZeppelin imports**
   - In Remix, create files and import:
     ```solidity
     import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
     import "@openzeppelin/contracts/access/Ownable.sol";
     import "@openzeppelin/contracts/token/ERC20/IERC20.sol";
     ```

2. **Deploy `RentToken`**
   - Compile `RentToken.sol`.
   - Deploy with constructor parameters:
     - `name_`: `Rental Income Token`
     - `symbol_`: `RNT`
     - `initialSupply_`: e.g. `1000000e18`
   - Record the deployed `RentToken` contract address.

3. **Deploy / identify `RewardToken`**
   - Option A: Deploy a simple ERC‑20 as test reward token (e.g. `TestUSD`).
   - Option B: Use an existing test token on the chosen testnet.

4. **Deploy `RentDistributor`**
   - Compile `RentDistributor.sol`.
   - Deploy with:
     - `_rentToken`: address of `RentToken`
     - `_rewardToken`: address of reward token (e.g. test stablecoin)
   - Record the deployed `RentDistributor` contract address.

5. **Basic test flow**
   - From the issuer address:
     1. Hold the initial RNT supply.
     2. Transfer some RNT to another address (simulating an investor).
   - Fund the distributor:
     1. Transfer some `rewardToken` to the `RentDistributor` contract address.
     2. Call `notifyRewardAmount(amount)` from the owner address.
   - From the investor address:
     1. Call `earned(investor)` to view expected payout.
     2. Call `getReward()` and verify that `rewardToken` balance increases.
